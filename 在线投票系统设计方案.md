# 在线投票系统设计方案

## 1. 需求目标

系统目标是实现一个可在 Kubernetes 中运行的在线投票系统，满足以下要求：

- 使用 Go 开发后端服务。
- 拆分 HTTP 网关服务和 gRPC 业务服务。
- 用户通过 Web 页面发起 HTTP 请求。
- HTTP 服务通过 gRPC 调用业务服务。
- 业务服务使用 Redis 存储投票数据。
- Web 页面实时展示投票结果。
- Kubernetes 中 `httpserver` 和 `grpcserver` 至少 3 个 Pod。
- 具备并发一致性测试和基础可观测性。

## 2. 当前部署方案

当前采用本机 kind Kubernetes 集群进行部署和演示。

```text
Docker Desktop -> kind 三节点集群 -> Kubernetes Deployment/Service -> Prometheus -> Grafana
```

当前部署方案不包含 Rancher，也不包含云服务器部署。Rancher 和云服务器部署可作为后续扩展，不作为当前项目交付路径。

## 5. Redis 数据结构

### 5.1 投票基础信息

```text
poll:{poll_id}
```

类型：Hash

字段：

| 字段 | 说明 |
|---|---|
| id | 投票 ID |
| question | 投票问题 |
| created_by | 创建者 |
| updated_by | 更新者 |
| created_at | 创建时间 |
| expires_at | 过期时间 |
| is_closed | 是否关闭 |
| is_public | 是否公开 |

### 5.2 投票选项

```text
poll:{poll_id}:options
```

类型：Hash

说明：field 为选项序号，value 为选项文本。

### 5.3 投票结果

```text
poll:{poll_id}:votes
```

类型：Hash

说明：field 为选项文本，value 为票数。

### 5.4 已投票用户

```text
poll:{poll_id}:voters
```

类型：Set

说明：记录已经参与该投票的用户 ID，防止重复投票。

### 5.5 用户投票历史

```text
user:{user_id}:votes
```

类型：Hash

说明：记录用户对各投票的选择。

### 5.6 幂等键

```text
idem:{idempotency_key}
```

类型：String，带 TTL

说明：用于避免客户端重试导致重复计票或重复撤销。

## 6. 并发一致性设计

投票和撤销都不是单个 Redis 命令，而是多个状态的组合更新。

投票需要同时完成：

- 判断用户是否已经投票。
- 增加选项票数。
- 写入已投票用户集合。
- 写入用户投票历史。
- 写入幂等键。

撤销需要同时完成：

- 判断用户是否已投票。
- 获取用户已投选项。
- 扣减对应选项票数。
- 删除已投票用户记录。
- 删除用户投票历史。
- 写入幂等键。

如果这些步骤分散执行，在高并发下可能出现重复计票、票数为负、用户记录和票数不一致等问题。因此 Redis Store 使用 Lua 脚本把相关步骤放在 Redis 单线程执行上下文内完成，保证单次业务操作原子性。

## 7. 幂等设计

客户端可以在投票或撤销请求中携带：

```text
Idempotency-Key: <唯一键>
```

服务端使用 Redis 记录幂等键。

效果：

- 同一个用户用同一个幂等键重复投票，不会重复增加票数。
- 同一个用户用同一个幂等键重复撤销，不会重复扣票。
- 网络失败或客户端重试时，可以降低重复提交风险。

### 13.2 微服务设计

系统拆分为两个无状态业务服务和一个 Redis 存储组件。

`httpserver`：

- 面向浏览器和 Web 前端提供 HTTP API。
- 使用 Gin 处理路由、中间件、静态资源和 SSE。
- 不保存投票状态，不直接访问 Redis。
- 通过 gRPC 调用 `grpcserver`。
- 在 Kubernetes 中部署 3 个 Pod，通过 Service 暴露为 NodePort `30080`。

`grpcserver`：

- 提供内部 gRPC API，承载核心投票业务。
- 调用 service 层完成参数校验、权限判断、业务规则处理。
- 调用 Redis store 完成投票数据读写。
- 不保存本地状态，支持多副本扩缩容。
- 在 Kubernetes 中部署 3 个 Pod，仅通过 ClusterIP 在集群内部访问。

Redis：

- 存储投票元数据、选项、票数、已投票用户集合、用户投票历史和幂等键。
- 投票和撤销操作通过 Lua 脚本保证原子性。
- 当前课程项目使用单实例 Redis，满足功能演示和一致性验证。

调用链路：

```text
Web 前端
  -> httpserver HTTP API
  -> grpcserver gRPC API
  -> Redis Lua 原子脚本
  -> 返回最新投票结果
```

多副本设计：

- `httpserver` 和 `grpcserver` 均为无状态服务，可以水平扩缩容。
- 请求无论进入哪个 `httpserver`、路由到哪个 `grpcserver`，最终都由 Redis 统一判断和计票。
- 服务副本变化只影响短暂可用性，不改变 Redis 中的最终一致性结果。

### 13.3 关键技术点

#### 13.3.1 Redis Lua 保证并发一致性

投票不是单个计数命令，而是多个状态的组合更新。一次投票必须同时完成：

- 判断用户是否已经投票。
- 判断选项是否合法。
- 增加对应选项票数。
- 写入已投票用户集合。
- 写入用户投票历史。
- 写入幂等键。

如果这些步骤由多条 Redis 命令分散执行，在高并发下可能出现两个请求同时判断用户未投票，然后都加票的问题。

本项目把投票和撤销逻辑写入 Redis Lua 脚本，在 Redis 单线程执行上下文中完成，保证检查和更新不可被其他请求插入。

#### 13.3.2 幂等键处理客户端重试

客户端可以携带：

```text
Idempotency-Key: <唯一请求键>
```

服务端将用户 ID 和幂等键组合记录到 Redis。对于同一用户的同一个逻辑请求，多次提交只产生一次业务影响。

该设计用于处理：

- 客户端超时后重试。
- HTTP/gRPC 响应返回途中连接中断。
- `grpcserver` 滚动重启期间请求失败后重试。

#### 13.3.3 无状态服务支持扩缩容和故障恢复

`httpserver` 和 `grpcserver` 不保存投票状态。服务扩容、缩容或单 Pod 宕机时：

- 请求未到达 Redis：不会计票，客户端可以重试。
- Redis Lua 已执行成功：数据已经提交，客户端用同一幂等键重试不会重复计票。
- 多个服务副本同时处理同一用户请求：Redis Lua 只会让其中一次有效计票成功。

因此，服务层扩缩容或宕机可能造成短暂失败，但不会破坏“不多投、不漏投”的一致性边界。前提是 Redis 可用且数据未丢失。

#### 13.3.4 SSE 实时刷新投票结果

Web 页面通过 SSE 接收结果刷新事件。投票、撤销、关闭、删除等操作完成后，`httpserver` 会通知前端重新拉取最新结果。

当前 SSE 通知为进程内广播，单个 `httpserver` Pod 内实时刷新正常；多 Pod 严格跨实例广播可通过 Redis Pub/Sub 继续增强。

#### 13.3.5 Prometheus + Grafana 可观测性

系统暴露以下指标：

```text
vote_http_requests_total
vote_http_request_duration_seconds
vote_grpc_requests_total
vote_grpc_request_duration_seconds
vote_grpc_errors_total
```

Prometheus 通过 Pod 注解抓取 `httpserver` 和 `grpcserver` 指标。Grafana 自动配置 Prometheus 数据源和 `Vote System Observability` 看板，用于展示高并发下的请求速率、延迟、错误速率和 Go goroutine 数。

#### 13.3.6 测试证明一致性

一致性测试分为两层：

- Redis Store 并发测试：直接验证 Lua 脚本在并发写入下不多投、不漏投。
- HTTP E2E 测试：从真实 HTTP 入口发请求，验证完整链路 `httpserver -> grpcserver -> Redis` 的最终状态。

### 13.4 开源项目参考

本项目的 Redis 设计参考了常见开源系统中使用 Redis Set、Hash 和原子脚本处理去重、计数和状态一致性的方式。

参考项目一：Redis 官方文档和示例

- Redis Hash 适合存储对象字段和计数值。
- Redis Set 适合记录唯一成员，例如已投票用户集合。
- Redis Lua 脚本适合把多步读写合并为一个原子操作。

本项目借鉴点：

- 使用 Hash 保存投票元数据和选项票数。
- 使用 Set 保存已投票用户，判断用户是否重复投票。
- 使用 Lua 脚本把检查和更新合并执行。

参考项目二：go-redis 官方示例和常见限流/幂等实现

- 使用 Redis key 加 TTL 保存短期状态。
- 使用 `SETNX`、`SETEX` 或 Lua 脚本实现幂等和去重。
- 使用 Go Redis 客户端封装存储访问，业务层不直接操作 Redis 命令。

本项目借鉴点：

- 使用带 TTL 的幂等 key 记录客户端重试请求。
- 使用 `github.com/redis/go-redis/v9` 封装 Redis 访问。
- 将 Redis 实现收敛在 `internal/store/redis`，service 层通过接口使用存储。

## 14. 已知限制

- Redis 当前是单实例，不提供高可用。
- Prometheus 和 Grafana 当前是轻量单实例，不提供高可用和持久化存储。
- 多 `httpserver` Pod 下 SSE 事件只在当前 Pod 内广播；若要严格支持跨 Pod 实时刷新，需要引入 Redis Pub/Sub 或消息队列。
- 当前使用 NodePort 暴露 Web，未配置 Ingress Controller。

## 15. 后续改进

- 为 SSE 增加 Redis Pub/Sub，支持跨 Pod 广播。
- 为 Redis 增加持久化 PVC 或使用外部 Redis。
- 增加告警规则和 Grafana 持久化存储。
- 增加浏览器级 E2E 测试，例如 Playwright 测试真实页面交互。
